# Mordor event aggregation sensitivity

This note records the first ASTRA sensitivity check using aggregated
Mordor-derived telemetry.

## Purpose

Earlier Mordor belief/usefulness runs showed that the raw event-level bridge
quickly saturates scalar host compromise belief. Most healthy-mode events were
therefore labelled redundant.

This experiment checks whether aggregating repeated Mordor-derived events into
host/window/event/state evidence units changes the belief/usefulness
diagnostics.

## Script

Generated by:

```bash
python scripts/run_mordor_event_aggregation_sensitivity.py
```

Output:

- `outputs/tables/mordor_event_aggregation_sensitivity.csv`

The CSV output is reproducible and is not tracked.

## Aggregation rule

The aggregation helper groups generated telemetry by:

```text
host_id
aggregation_window_start
aggregation_window_end
event_type
source_state
```

For each group, it emits one ASTRA-compatible generated telemetry row.

The aggregated event uses:

- maximum event score as `event_score`;
- logical any over `is_true_signal`;
- `source_event_count` to preserve how many raw records were represented;
- deterministic aggregate event IDs.

Aggregation is applied before impairment and belief update.

## Results

| Mode | Window | Generated events | Source event count | Useful | Redundant | Misleading | Saturated update fraction |
|---|---:|---:|---:|---:|---:|---:|---:|
| raw | — | 6,026 | 6,026 | 7 | 5,986 | 33 | 0.993 |
| aggregated | 5 | 44 | 6,026 | 0 | 18 | 26 | 0.409 |
| aggregated | 10 | 44 | 6,026 | 0 | 18 | 26 | 0.409 |
| aggregated | 30 | 44 | 6,026 | 0 | 18 | 26 | 0.409 |

## Interpretation

Aggregation substantially changes the unit of evidence. The raw bridge treats
6,026 Mordor-derived records as 6,026 belief-update opportunities. The
aggregated bridge represents the same 6,026 source records as 44 evidence units.

This sharply reduces the saturated-update pattern. The saturated update
fraction falls from 0.993 in the raw event-level bridge to 0.409 in the
aggregated bridge.

However, aggregation does not automatically produce more useful updates under
the current state mapping and usefulness diagnostic. Instead, the aggregated
runs produce 26 misleading updates and 18 redundant updates, with no useful
updates.

The identical results for 5-, 10-, and 30-step windows suggest that the Mordor
event burst structure is compact enough that these window sizes produce the
same grouping under the current aggregation keys.

## Main point

Aggregation solves one problem but exposes another.

It reduces repeated saturated updates by changing the unit of evidence, but it
also makes the current `observed_state` to `source_state` abstraction more
visible. Under the current diagnostic, many aggregated evidence units increase
compromise belief when the delivery-time state abstraction is not compromised,
so they are labelled misleading.

This suggests that the next modelling issue is not just event volume. It is the
state abstraction used to judge belief usefulness.

## Interpretation boundary

This result should not be interpreted as saying the aggregated Mordor alerts
are operationally misleading. It says that under the current ASTRA state
mapping and belief-error diagnostic, those aggregated updates increase belief
in a way that does not match the delivery-time state abstraction.

The raw event bridge should remain the baseline. Aggregation should remain an
optional sensitivity path until the state abstraction is improved.
